# 開発計画（plan.md）

このドキュメントは `docs/memo.txt` に記載された要望をもとに、
脆弱性診断メモ整理ツールの開発計画をまとめたものである。

参照：`docs/memo.txt`、`AGENTS.md`

## 1. 目的

ウェブサイトの脆弱性診断中に記録したメモを整理し、
再現手順・影響・証跡・対策案を分かりやすくまとめるためのツールを作る。

散らばりがちな診断メモを統一フォーマットで管理し、
最終的な報告書作成を効率化することを目的とする。

## 2. 前提・方針（memo.txtより）

* 日本語ユーザー向けに作る（画面文言・メッセージはすべて日本語）。
* 自分専用ツールとして利用する。共有機能・マルチユーザー機能は考えない。
* 最初はシンプルな構成にする。カスタマイズ項目は今後少しずつ増やす。
* 設定は誰にでも分かりやすいものにする（設定項目を増やしすぎない）。
* AGENTS.mdの方針（機密情報のマスク、攻撃自動化の禁止など）を踏襲する。

## 3. 想定ユーザー

* このツールを作った本人（自分1人）。
* 学習・許可された検証環境での脆弱性診断メモ整理を目的とする。

## 4. 技術スタック

シンプルさを優先し、個人利用・学習用途に適した構成にする。

| 項目 | 選定 |
|---|---|
| 言語 | Python |
| Webフレームワーク | Flask（軽量でシンプルなため） |
| テンプレートエンジン | Jinja2（Flask標準） |
| データ保存 | SQLite（ファイル1つで完結し、個人利用に十分） |
| ORM / DBアクセス | 標準の`sqlite3`モジュール（最初はシンプルに。必要になればSQLAlchemy検討） |
| フロントエンド | 素のHTML + CSS + 最小限のJavaScript（フレームワーク不要） |
| 認証 | なし（自分専用・ローカル利用のため。将来的に必要なら簡易パスワード程度を検討） |

外部サービスへの通信・自動攻撃・スキャン機能は実装しない。

## 6. データモデル（案）

AGENTS.md記載の項目に基づき、1件の脆弱性メモを以下の項目で管理する。

| フィールド名 | 説明 |
|---|---|
| id | メモのID（主キー、自動採番） |
| title | 脆弱性名（例：IDOR、認可不備など） |
| target_url | 対象URL |
| http_method | HTTPメソッド |
| request_body | リクエスト内容（マスク前の生データも保持するかは要検討） |
| response_body | レスポンス内容 |
| summary | 概要（1〜3文） |
| reproduction_steps | 再現手順 |
| observed_result | 確認結果 |
| impact | 影響 |
| severity | 深刻度（High / Medium / Low / Info） |
| severity_reason | 深刻度の判断理由 |
| evidence_note | 証跡メモ |
| fix_suggestion | 対策案 |
| status | 確認状況（例：未着手 / 確認中 / 確認済み / 要再確認） |
| remarks | 備考（未確認事項など） |
| created_at | 作成日時 |
| updated_at | 更新日時 |

* `severity` と `status` は選択式（プルダウン）にして入力揺れを防ぐ。
* 保存前に `mask.py` でCookie・パスワード・トークン等をマスクしてから保存する
  （マスク方針はAGENTS.mdの「セキュリティ上の注意」に準拠）。

## 7. 主な機能（MVP：最初に作る範囲）

1. メモ一覧表示（脆弱性名・対象URL・深刻度・確認状況を一覧表示）
2. メモ新規作成（フォーム入力 → マスク処理 → SQLiteに保存）
3. メモ編集・削除
4. メモ詳細表示（`docs/memo-format.md`に定義した標準フォーマットで表示）
5. 入力値のHTMLエスケープ表示（XSS対策）
6. 機密情報マスク処理（保存時・表示時・エクスポート時）
7. 報告書用まとめのエクスポート（Markdown形式でのコピー・出力）

## 8. 今後の拡張候補（MVP後）

memo.txtの「カスタマイズは今後増やしていく」という方針に沿って、
以下は後回しにする候補とする。

* 検索・絞り込み（深刻度別、確認状況別など）
* タグ・カテゴリ管理
* スクリーンショット添付
* CSV / PDFなど他形式でのエクスポート
* メモのテンプレートを複数用意して選べるようにする
* ダークモードなどの表示設定

## 9. 画面構成（案）

* `/memos`：メモ一覧画面
* `/memos/new`：新規作成画面
* `/memos/<id>`：詳細画面
* `/memos/<id>/edit`：編集画面
* `/memos/<id>/export`：報告書用まとめ表示・コピー画面

## 10. セキュリティ・実装時の注意（AGENTS.mdより再掲）

* 入力値は表示時に必ずHTMLエスケープする（メモ本文にHTML/JSが含まれても実行させない）。
* Cookie、セッションID、Authorizationヘッダー、パスワード、APIキー、
  メールアドレス、電話番号、個人名、住所、学籍番号・社員番号は
  保存・表示・エクスポートのいずれの場面でもマスクする。
* エクスポート機能はマスク済みデータのみを出力する。
* 診断対象への自動攻撃・スキャン機能は実装しない。
* ローカル保存（SQLiteファイル）であっても機密情報の扱いには注意する
  （`.gitignore`でリポジトリに含めない）。

## 11. 開発の進め方（フェーズ分け）

1. **フェーズ1：土台作り**
   Flaskアプリの雛形作成、SQLite初期化、基本ルーティング確認。
2. **フェーズ2：メモのCRUD実装**
   一覧・新規作成・詳細・編集・削除を一通り動かせるようにする。
3. **フェーズ3：マスク処理・エスケープ処理の実装**
   保存前マスク、表示時エスケープを組み込み、動作確認する。
4. **フェーズ4：エクスポート機能**
   `skills/report-export-helper`の出力フォーマットに沿ったMarkdown出力を実装する。
5. **フェーズ5：見た目・使い勝手の調整**
   シンプルなCSSでの整形、入力しやすいフォームレイアウトの調整。
6. **フェーズ6以降：拡張機能**
   「9. 今後の拡張候補」から必要なものを順次追加する。

各フェーズ完了時に、AGENTS.md記載の通り
「入力・表示・保存・エクスポートの基本動作」を確認する。

## 12. 決定事項（旧・未確定事項）

以下は当初「未確定事項」としていたが、方針を確定した。

* **生データの保存**：リクエスト/レスポンスの生データ（マスク前）は保存しない。
  マスク後のデータのみをDBに保存する。これにより機密情報が残るリスクを避ける。
* **ORM**：SQLAlchemyなどは導入せず、標準の`sqlite3`モジュールのみで実装する。
  個人利用の規模であれば十分であり、学習コストも抑えられる。
* **認証機能**：ログイン・パスワードロック機能は追加しない。
  自分専用のローカル利用が前提のため不要と判断した。
