# VulnNote Manager 開発タスクリスト

## 1. この文書の目的

この文書は、`docs/spec.md` バージョン0.1.0を初版として実装し、受け入れ条件を満たすまでのタスクリスト兼実装計画である。

- タスクは原則として上から順に進める。
- 各フェーズの「完了条件」を満たしてから、依存する次フェーズへ進む。
- 機能実装と同じフェーズで自動テストを追加し、後工程へテストを先送りしない。
- 仕様にない事項や仕様変更が必要な事項は推測で確定せず、利用者確認後に `docs/spec.md` と本書を更新する。
- 初版の対象外機能は「19. 初版では実施しない項目」に従い、初版へ混在させない。

## 2. タスクの見方と進め方

### 2.1 ステータス

- `[ ]`: 未着手または作業中
- `[x]`: 実装、レビュー、必要なテストまで完了

### 2.2 優先度

| 優先度 | 意味 |
| --- | --- |
| P0 | 後続工程を止める前提、または安全性・データ整合性に直結する必須項目 |
| P1 | 初版の受け入れ条件を満たすための必須機能 |
| P2 | 品質、操作性、保守性を完成水準へ引き上げる必須項目 |

### 2.3 共通の完了定義（Definition of Done）

各実装タスクは、次をすべて満たした時点で完了とする。

- 仕様に定めた正常系、異常系、境界値を実装している。
- サーバー側で入力値を検証し、エラーは原因と対処が分かる日本語で表示する。
- データ変更処理にCSRF対策を適用し、SQLはプレースホルダーを使用する。
- HTML出力は既定でエスケープし、秘密情報やメモ本文をログへ記録しない。
- 公開関数と複雑な処理に型ヒントおよび必要最小限の説明を付ける。
- 対応する単体テストまたは結合テストを追加し、テスト一式が成功する。
- キーボード操作、フォーカス、ラベル、色以外の状態表現を確認する。
- 関連するREADME、仕様、運用手順に変更があれば同じ変更で更新する。

## 19. 初版では実施しない項目

次は将来候補として管理し、別途仕様と優先度が確定するまで初版タスクへ追加しない。

- 複数利用者、ログイン認証、権限管理、操作履歴
- LANまたはインターネットへの公開運用
- バックアップ・復元
- Dockerによる起動・配布
- ゴミ箱・削除データ復元
- 脆弱性メモの複製
- PDF・DOCX出力、テンプレート管理
- CVSSスコア計算
- モバイルアプリ
- 外部チケット管理、診断ツール、クラウドストレージ連携
- 記録漏れを指摘するAI補助
- AIプロバイダー切り替え抽象化

## 20. 変更管理

- [ ] 仕様変更時は、変更理由、影響する受け入れ条件、データ移行要否を記録する。
- [ ] `docs/spec.md` を先に更新し、本書のタスク、優先度、依存関係、テストを同期する。
- [ ] 完了済みタスクへ影響する変更では、該当タスクを未完了へ戻して再検証する。
- [ ] 新しい外部依存を追加する場合は、目的、必要性、標準ライブラリによる代替案、保守・セキュリティ影響を記録する。
