# テストとリリース確認

## 自動テスト

標準コマンドは `python -m pytest` である。すべてのテストはpytestの一時ディレクトリへSQLiteと画像領域を作り、利用者の保存先や実OpenAI APIを使わない。

- `tests/test_config.py`: OS別保存先、設定境界、秘密情報を含まない設定
- `tests/test_database.py`: マイグレーション、制約、外部キー、ロールバック、競合
- `tests/test_validation_security.py`: 入力検証、CSRF、HTMLエスケープ、HTTPヘッダー
- `tests/test_catalog.py`: 3階層CRUD、検索、絞り込み、ページング、並べ替え
- `tests/test_services.py`: 削除ロック、画像検証・配信・削除、CSV／Markdown安全化
- `tests/test_extended_features.py`: 複数削除、条件一致全件削除、画像一括ロールバック、ZIP Slip対策
- `tests/test_ai.py`: 機密情報警告、Responses API呼び出し形状、同意、比較、選択採用

実API試験は通常テストへ混在させない。実施する場合は、送信内容を架空データだけに限定し、別途 `real_api` マーカー付きテストとして追加する。

## 手動確認が必要な項目

リリース判定前にWindows、macOS、LinuxおよびChrome、Edge、Firefox、Safari最新版で次を確認する。

- READMEどおりの新規インストール、起動、終了、OS別保存先
- キーボードだけでの全主要操作、フォーカス順、ラベルとエラーの関連付け
- 文字・背景・フォーカスのコントラスト、色以外の状態表現
- 長いURL・連続文字・証跡、複数画像、狭い画面でのレイアウト
- CRUD、削除確認、画像表示、Markdown／CSV／ZIPダウンロード
- 架空データを使ったAI送信確認、取消、結果不採用、報告書出力

実機確認結果は日付、OS／ブラウザのバージョン、確認者、結果、不具合IDを記録する。未実施の環境を推測で合格扱いしない。

確認結果は [`release-checklist.md`](release-checklist.md) に記録する。

## セキュリティ上の不変条件

- すべての変更リクエストにCSRF検証がある。
- SQL値はプレースホルダーを使い、並べ替え識別子は固定辞書から選ぶ。
- 利用者本文はJinjaで自動エスケープし、ログへ記録しない。
- 画像は公開静的ディレクトリへ置かず、検証済みMIMEタイプでアプリ経路から配信する。
- 親削除は自身と全配下のロックを実行時に再評価する。
- AIは確認画面に表示したテキストだけを明示同意後に送信し、画像を送信しない。
- AI失敗、不採用、競合時に保存済みデータを変更しない。

## 仕様変更の運用

仕様変更では、変更理由、影響する受け入れ条件、データ移行要否を先に整理して `docs/spec.md` を更新する。その後、`docs/tasks.md` の優先度・依存関係・完了状態と対応テストを同じ変更で同期する。完了済み機能へ影響する場合は該当タスクを未完了へ戻し、再検証する。外部依存を追加するときは、目的、必要性、標準ライブラリ案、保守・セキュリティへの影響を `docs/design.md` に記録する。
